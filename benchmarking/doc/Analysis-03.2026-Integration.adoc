== Executive Summary

WRK stress testing (5 threads, 50-300 connections, 0% error rate) on native Quarkus with virtual threads:

* **JWT Validation**: 25,178 ops/s at 50 connections (2.30ms avg), exceeds 10K target
* **Health Check**: 98,211 ops/s at 50 connections (0.78ms avg)
* **JWT/Health ratio**: stable ~3x across all connection counts — JWT validation under concurrency is the dominant cost (link:Analysis-03.2026-Latency-Decomposition.adoc[decomposition])
* **Graceful degradation**: JWT sustains 22,745 ops/s at 300 connections

== Configuration

* **Tool**: WRK (HTTP benchmarking tool with Lua scripting)
* **Profile**: Stress (5 threads, variable connections)
* **Duration**: 1 minute per benchmark
* **Environment**: 10 CPU cores (M4), Docker with native executable
* **Token management**: Maven-managed infrastructure fetches fresh tokens per benchmark run

=== Key Observations

* **50 connections is optimal**: peak throughput for all endpoints
* **Smooth monotonic degradation** from 50 to 300 connections across all endpoints
* **JWT/Health latency ratio stable** at ~3.0-3.7x across all connection counts, confirming proportional scaling
* **JWT and Direct Validation track identically** at every connection count — the gap never exceeds 210µs (<2%), confirming CDI overhead is noise (see <<_cdi_overhead>>)
* **At 300 connections**, JWT at 22,745 req/s with 58.52ms P99

== Aspects Evaluated but Ruled Out

This section consolidates factors investigated as potential causes of the health-to-JWT gap, with evidence showing they are not significant.

=== Lock Contention

JFR recorded only 14 `JavaMonitorEnter` events over 240 seconds — sporadic Vert.x connection write queue contention, not a bottleneck.

=== Garbage Collection

GC accounts for ~3.7% of wall-clock time (1,312 events in 60s, 22 GCs/second). At ~25K req/s, only 1 in 1,140 requests encounters a GC pause. GC contributes to P99 tail latency but not median latency. See link:Analysis-03.2026-JFR-Profiling.adoc#_garbage_collection_analysis[JFR GC analysis] for raw data.

=== Access Log Filter

The `CustomAccessLogFilter` runs on all endpoints. Disabling it improved health P50 by ~0.1ms but had no effect on JWT. The filter does not explain the JWT-health gap.

=== CDI `@ApplicationScoped` Injection

The ablation study confirmed zero per-request cost — a bare JAX-RS endpoint without CDI was actually 0.086ms slower than the CDI-injected variant (benchmark ordering variance).

[[_cdi_overhead]]
=== CDI `Instance.get()` Overhead

The `JwtValidationEndpoint` uses `Instance<BearerTokenResult>.get()`, a `@Dependent`-scoped CDI producer. Each call creates a `CreationalContext`, performs `InjectionPoint` introspection, and invokes the producer.

The ablation study measured this overhead at **~20µs** (JWT avg 2.30ms minus Direct Validation avg 2.28ms). This is within measurement noise and **not a meaningful performance factor**.

Quarkus ArC's `@WithCaching` could eliminate this overhead, but **cannot be applied** because the endpoint uses 5 different `Instance` fields with different `@BearerToken` annotation parameters (different required scopes/roles/groups). Caching would return incorrect results — a security bypass. The `@Dependent` scope is semantically required. See link:Analysis-03.2026-JFR-Profiling.adoc#_cdi_producer_chain_research_findings[JFR CDI research] for details.

=== `@RequestScoped` Producer Refactoring

Not feasible — multiple injection points use different `@BearerToken` parameters, so a request-scoped cache would break security validation.

== Root Cause: CPU Saturation From RSA Under Concurrency

After ruling out all framework-level factors, the dominant cost is **CPU saturation from RSA signature verification under concurrency**.

=== Evidence

RSA verification is compute-bound (88% of the validation pipeline). With 114 avg threads (JFR) competing for 10 CPU cores at 85% peak utilization, this is **~11:1 thread-to-core oversubscription**. Most threads are waiting for CPU time, not executing.

A JMH concurrency sweep on HotSpot JVM confirms this is **not a native-image issue**:

[cols="1,2,2", options="header"]
|===
| Threads | Avg Latency | Multiplier

| 1 | 48µs | 1.0x
| 10 | 89µs | 1.9x (fits 10 cores — nearly linear)
| 50 | 556µs | 11.6x (5:1 oversubscription)
| 100 | 957µs | 20.0x (10:1 oversubscription)
|===

The 1→10 thread scaling is nearly linear (10 threads on 10 cores). Beyond 10 threads, latency degrades proportionally to oversubscription. The native integration cost of 1,060µs at 50 connections adds ~2x on top of the JVM baseline (556µs) — consistent with HTTP/TLS/Quarkus overhead.

=== Why CPU% Doesn't Match Latency%

JFR shows the OAuth library at 0.3% of CPU samples, yet the ablation study measures JWT validation at 46.1% of wall-clock time. CPU saturation explains this: JFR samples **on-CPU** time, but with ~11:1 oversubscription most threads are **off-CPU** (waiting to be scheduled) at any given sample point. The wall-clock cost includes both on-CPU and queuing time.

=== Implications

The lock-free cache design is already optimal (14 lock events over 240 seconds). The bottleneck is fundamental: RSA 2048-bit signature verification is compute-bound, and the thread count exceeds available CPU cores. No further optimization path exists within the current architecture short of reducing thread count or switching to a less compute-intensive algorithm (not benchmarked).

== Conclusion

* **25.2K ops/s** at 50 connections exceeds the 10K target by 2.5x
* **2.30ms avg** latency well under the 5ms target
* **CPU saturation from RSA under concurrency** (1,060µs, 46.1% of request time) is the sole significant cost — all framework overhead is negligible
* **Not native-image-specific**: JMH concurrency sweep shows the same degradation on HotSpot JVM (556µs at 50 threads)
* **No optimization path** within the current architecture — bottleneck is thread-to-core oversubscription for compute-bound RSA
* **Stable scaling** through 300 connections with no anomalous behavior
