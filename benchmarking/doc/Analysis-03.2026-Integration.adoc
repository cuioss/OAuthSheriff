== Executive Summary

Performance testing using WRK with stress profile (5 threads, 50-300 connections) shows:

* **JWT Validation (cache enabled, size 20)**: 25,178 ops/s at 50 connections (peak JWT performance)
* **Health Check**: 98,211 ops/s at 50 connections
* **Direct Validation**: 24,822 ops/s at 50 connections
* **Mock JWT (CDI bypass)**: 54,748 ops/s at 50 connections
* **Ablation Baseline**: 57,054 ops/s at 50 connections
* **Ablation Header-Only**: 53,845 ops/s at 50 connections
* **Target Achievement**: Exceeds minimum target (>10k ops/s) at 25.2k ops/s
* **Error Rate**: 0% across all connection counts (50-300), no token expiration — Maven-managed benchmark infrastructure fetches fresh tokens per benchmark run
* **CDI Contention**: Negligible across all connection counts (20µs at 50 connections, within noise through 300 connections)

=== Latency Decomposition

A six-layer ablation study decomposes the full JWT avg latency (2.30ms) at 50 connections into measured layers. The dominant cost is JWT validation under concurrency at 1,060µs (46.1%), measured via the direct-validation ablation variant (which bypasses CDI `Instance.get()`). CDI `Instance.get()` + `@Timed` adds only 20µs (0.9%) at 50 connections. Shared infrastructure accounts for 0.78ms (33.9%), JAX-RS POST routing contributes 0.40ms (17.4%), and header resolution adds 0.06ms (2.6%). CDI `@ApplicationScoped` injection has zero per-request cost.

The Maven-managed benchmark infrastructure eliminates token expiration artifacts, providing clean data across all connection counts. CDI overhead remains negligible (~20µs) through 300 connections with no divergence at any connection count.

Full methodology, variant descriptions, raw benchmark data, and analysis: link:Analysis-03.2026-Latency-Decomposition.adoc[Latency Decomposition — Ablation Study].

==== Scaling Observation

At higher connection counts, latency grows due to connection queuing. The JWT/Health avg latency ratio remains stable (~3.0-3.7x) across all connection counts (50 conns: 2.95x, 100 conns: 3.56x, 200 conns: 3.70x, 300 conns: 3.60x), confirming proportional scaling. At all connection counts, the JWT/Health and Direct Validation/Health ratios track closely (2.95x vs 2.92x at 50 conns; 3.56x vs 3.51x at 100 conns; 3.60x vs 3.57x at 300 conns), confirming CDI overhead is negligible across the entire range.

=== Performance Ceiling

* **Health check capacity**: 98,211 ops/s at 50 connections (peak performance)
* **JWT validation capacity**: 25,178 ops/s at 50 connections (peak JWT performance, cache enabled, 100% hit rate)
* **Direct Validation capacity**: 24,822 ops/s at 50 connections
* **Mock JWT capacity**: 54,748 ops/s at 50 connections (CDI bypass, no JWT validation)
* **Ablation Baseline capacity**: 57,054 ops/s at 50 connections (bare JAX-RS + CDI)
* **Performance gap**: ~73,033 ops/s between health and JWT endpoints at 50 connections
* **Graceful degradation**: JWT sustains 22,745 ops/s even at 300 connections, exceeding 10K target

**Throughput gap explanation:**

At 50 connections, the ~3.9x throughput difference (98.2K health vs 25.2K JWT) is explained by the ~2.95x per-request latency ratio (0.78ms health vs 2.30ms JWT avg). With a fixed number of connections, throughput is inversely proportional to per-request latency. The ratio remains stable across all connection counts (3.0-3.7x), confirming proportional scaling.

== Conclusion

Comprehensive WRK stress testing across 50-300 connections with Maven-managed benchmark infrastructure reveals:

* **Peak JWT performance**: 25,178 ops/s at 50 connections (cache enabled, 100% hit rate)
* **Peak health performance**: 98,211 ops/s at 50 connections
* **0% error rate across all connection counts**: Maven infrastructure fetches fresh tokens per benchmark run, eliminating token expiration entirely
* **Graceful degradation**: JWT sustains 22,745 ops/s at 300 connections (15.11ms avg), comfortably exceeding the 10K target
* **CDI overhead negligible at ALL connection counts**: JWT and Direct Validation track within noise (0-210µs gap, <2% of avg) from 50 to 300 connections — no contention threshold observed
* **JWT/Health ratio stable**: ~3.0-3.7x across all connection counts, confirming proportional scaling without CDI contention as a factor
* **Six-endpoint ablation study**: Baseline/Header-Only/Mock JWT cluster at ~54K ops/s, Direct Validation and JWT cluster at ~25K ops/s — confirming JWT validation is the dominant cost
* **Cache effectiveness**: Lock-free cache achieves 100% hit rate (size 20), zero performance collapse
* **Smooth connection sweep**: All 6 endpoints show monotonic, predictable degradation from 50 to 300 connections with no anomalous data points

== Aspects Evaluated but Ruled Out

This section consolidates factors systematically investigated as potential causes of the ~14x JMH-to-integration JWT validation gap (77µs → 1,060µs under 50 concurrent connections), with measured evidence showing they do NOT explain the performance difference.

=== Lock Contention

**Finding**: Lock contention is **not** a performance factor.

[cols="2,3", options="header"]
|===
| Metric | Value

| JavaMonitorEnter events (240s benchmark) | 14 total
| Monitors involved | `Http1xServerConnection` (6), `EPollSelectorImpl` (8)
| Per-event duration | 10-14ms
| Impact on 25K req/s throughput | Negligible (<0.001%)
|===

The 14 lock events over 240 seconds represent sporadic Vert.x HTTP/1.x connection write queue contention — standard behavior, not a bottleneck.

=== Garbage Collection

**Finding**: GC overhead is **not** the primary cause of the ~14x gap.

[cols="2,3", options="header"]
|===
| Metric | Value

| GC events (60s benchmark) | 1,312
| Total pause time | 2,223ms (3.7% of wall-clock)
| GC frequency | 22 GCs/second
| Requests per GC | ~1,140 (25K req/s ÷ 22 GCs/s)
| Pause distribution | 89% at 1-2ms, 10% at >10ms
| Heap usage (sawtooth) | 10-100 MB, avg 45 MB
|===

At 22 GCs/second with ~25,000 requests/second, only 1 in 1,140 requests encounters a GC pause. The 3.7% overhead is amortized — most individual requests complete without GC interference. GC contributes to tail latency (P99) but not median latency.

=== Access Log Filter

**Finding**: The filter adds overhead but does **not** explain the JWT-health gap.

The `CustomAccessLogFilter` runs on **all** JAX-RS endpoints (both health and JWT), executing `shouldLog()` and `isPathIncluded()` on every response.

[cols="2,2,2,2", options="header"]
|===
| Endpoint | Filter Enabled | Filter Disabled | Difference

| Health P50 | 0.510ms | 0.417ms | **-0.093ms (18%)**
| JWT P50 | 1.91ms | 1.96ms | +0.05ms (variance)
| JWT/Health ratio | 4.55x | 4.70x | ~same
|===

**Conclusion**: Filter overhead is ~0.1ms per request, measurable on the health endpoint (0.510ms → 0.417ms) but within noise on the JWT endpoint. The JWT-health gap remains unchanged when the filter is disabled, confirming it does not explain JWT-specific latency.

=== CDI @ApplicationScoped Injection

**Finding**: CDI constructor injection has **zero** per-request cost.

The ablation study compared a bare JAX-RS endpoint (zero CDI) against a CDI-injected endpoint:

[cols="2,2,2", options="header"]
|===
| Variant | P50 (ms) | Notes

| A: Bare Baseline (no CDI) | 0.870 | Zero `@Inject` dependencies
| B: JAX-RS Baseline (CDI-injected) | 0.784 | `@ApplicationScoped` class
| Difference (A − B) | **+0.086ms** | CDI is _faster_
|===

Variant A (no CDI) was 0.086ms **slower** than B (CDI-injected) — the opposite of what CDI overhead would cause. This confirms `@ApplicationScoped` beans are resolved at startup, not per request. The 0.086ms difference is attributable to benchmark ordering variance (A ran 6th, B ran 4th).

=== @RequestScoped Producer Refactoring

**Finding**: Refactoring to `@RequestScoped` is **not feasible** due to semantic constraints.

The `BearerTokenProducer.produceBearerTokenResult()` method is `@Dependent` scoped. Investigation of a `@RequestScoped` refactoring revealed:

* Multiple injection points use different `@BearerToken` annotation parameters (different required scopes/roles)
* A `@RequestScoped` producer would cache the **first** call's result, breaking security validation for subsequent calls with different requirements
* The current design correctly re-evaluates requirements per injection point

**Conclusion**: The `@Dependent` scope is semantically required for correct security behavior.

=== Root Cause Analysis: Combined CDI Producer + JWT Validation Cost (Investigation Complete)

After systematically ruling out lock contention, GC, access log filter, CDI `@ApplicationScoped` injection, and `@RequestScoped` refactoring, the JWT-specific cost has been decomposed into two contributing factors. A direct-validation ablation variant (which calls `tokenValidator.createAccessToken()` directly, bypassing CDI `Instance.get()`) provides a **measured split**.

==== Contributing Factor 1: JWT Validation Under Concurrency (1,060µs, measured 46.1%)

The core JWT library validates in 77µs (JMH, single-thread isolation). Under 50 concurrent connections, the measured cost is 1,060µs (Mock JWT avg 1.22ms → Direct Validation avg 2.28ms), a ~14x increase due to:

* **CPU cache line invalidation** — concurrent access to shared lock-free cache
* **Memory barriers** — `ConcurrentHashMap` operations require ordering guarantees
* **Real object allocation** — `AccessTokenContent`, `HashMap` for claims, `Set` for scopes/roles/groups (vs JMH reusing pre-parsed data)
* **Thread scheduling** — 50 concurrent connections cause frequent context switches

==== Contributing Factor 2: CDI Instance.get() + @Timed Overhead (~20µs at 50 conns)

Each `Instance.get()` call in Quarkus ArC performs CreationalContext allocation, InjectionPointImpl construction, `@BearerToken` annotation introspection, bean instance creation, and `@Timed` Micrometer instrumentation. At 50 connections, the measured ablation data shows these operations add only ~20µs total (Direct Validation avg 2.28ms -> JWT avg 2.30ms). The CDI machinery is highly optimized in Quarkus ArC.

With Maven-managed benchmarks and fresh tokens, CDI overhead remains negligible through all connection counts tested (50-300). The JWT-DV gap stays within measurement noise (<210µs, <2%) at every connection count, confirming no CDI contention threshold exists in this range.

==== Why Neither Factor Can Be Optimized

**JWT concurrency penalty**: Inherent to concurrent workloads with real object allocation. The lock-free cache design is already optimal (14 lock events over 240 seconds).

**CDI `Instance.get()` overhead**: Quarkus ArC provides `@WithCaching` to cache `Instance.get()` results, but it **cannot be applied** here:

* The `JwtValidationEndpoint` uses 5 different `Instance<BearerTokenResult>` fields with different `@BearerToken` annotation parameters
* Each field requires different authorization validation (scopes, roles, groups)
* Caching would return the wrong result → **security bypass**

**Conclusion**: Both the `@Dependent` scope with per-request `InjectionPoint` introspection and the concurrent validation cost are **inherent to this design**. The combined overhead (1,060µs JWT validation + 20µs CDI) cannot be reduced without API changes.
