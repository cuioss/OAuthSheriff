== Overview

A **mock JWT endpoint** (`/mock-jwt/validate`) was created to directly measure the latency decomposition. The mock endpoint replicates the JWT endpoint's full HTTP processing path ‚Äî CDI injection, `HttpServletRequestResolver.resolveHeaderMap()`, `Authorization` header extraction, and `ValidationResponse` JSON serialization ‚Äî but **skips the actual JWT library validation** (`tokenValidator.createAccessToken()`). This turns the inferred decomposition into a measured one.

An **ablation study** further decomposes the gap between the health endpoint and mock JWT endpoint by progressively adding processing layers. Four additional endpoints isolate individual cost components:

=== Key Findings

**JAX-RS POST routing is a significant cost component.** The 0.40ms cost of JAX-RS POST routing + `@RunOnVirtualThread` dispatch + Jackson serialization accounts for 17.4% of the total JWT avg latency. This is expected ‚Äî POST requests with JSON serialization of a `ValidationResponse` record require more processing than the SmallRye Health GET endpoint.

**CDI `@ApplicationScoped` injection has zero per-request cost.** Historical Variant A comparison confirmed `@ApplicationScoped` beans are resolved at startup, not per request.

**Vert.x context resolution is minimal.** The 0.06ms cost of `resolveHeaderMap()` accounts for 2.6% of total latency. Authorization parsing adds no measurable overhead beyond this.

**JWT validation under concurrency is the dominant cost at 46.1% (1,060¬µs) ‚Äî measured, not estimated.** The Direct Validation variant (D) calls `createAccessToken()` directly, bypassing CDI `Instance.get()` and `@Timed` instrumentation. The 1.06ms gap between D and Mock JWT isolates the pure JWT validation cost under 50 concurrent connections ‚Äî ~14x slower than the 77¬µs JMH baseline due to cache contention, memory barriers, and real object allocation under concurrency.

**CDI `Instance.get()` + `@Timed` overhead is only 20¬µs (0.9%).** The 0.02ms gap between the full JWT endpoint and variant D measures the actual cost of CDI proxy resolution (`CreationalContext`, `InjectionPoint` introspection) plus Micrometer `@Timed` instrumentation. This is negligible compared to the JWT validation itself.

=== Ablation Variant Descriptions

* **A: Bare Baseline** (`/mock-jwt-bare/baseline`) ‚Äî A separate JAX-RS class with zero CDI constructor injection. Returned a pre-built `ValidationResponse`. Isolated JAX-RS + `@RunOnVirtualThread` + Jackson without any CDI beans.
* **B: JAX-RS Baseline** (`/mock-jwt/baseline`) ‚Äî Same behavior as A, but in the CDI-injected `MockJwtValidationEndpoint` class. Comparing A vs B confirms that `@ApplicationScoped` constructor injection has zero per-request cost.
* **C: Header Only** (`/mock-jwt/header-only`) ‚Äî Calls `resolveHeaderMap()` but does not parse headers or construct a HashMap. Returns a pre-built response. The difference C minus B isolates the Vert.x context resolution cost.
* **D: Direct Validation** (`/mock-jwt/direct-validate`) ‚Äî Calls `createAccessToken()` directly on the injected `TokenValidator`, bypassing CDI `Instance.get()` proxy resolution and `@Timed` instrumentation. The difference D minus Mock JWT isolates the pure JWT validation cost under concurrency, while JWT minus D isolates the CDI overhead.

== JMH vs Integration Validation Cost

The JMH micro-benchmark measures isolated JWT library validation at 0.077ms (77¬µs). The ablation study measures the actual JWT validation cost under concurrency at 1.06ms (variant D minus Mock JWT) ‚Äî **13.8x more** than the isolated library. This gap decomposes into two independently measured factors:

**Primary factor (46.1%, measured): JWT validation under concurrency = 1,060¬µs**

* **`tokenValidator.createAccessToken()` under concurrency** ‚Äî 50 concurrent connections competing for the lock-free cache, CPU cache line invalidation, memory barriers
* **`AccessTokenContent` construction** ‚Äî real object allocation from parsed claims (vs JMH reusing pre-parsed data)
* 13.8x concurrency multiplier over the 77¬µs JMH baseline

**Secondary factor (0.9%, measured): CDI `Instance.get()` + `@Timed` = 20¬µs**

* **CDI `Instance<BearerTokenResult>` proxy resolution** ‚Äî CDI creates `CreationalContext` and invokes the producer per request
* **`InjectionPoint` metadata extraction** ‚Äî CDI inspects `@BearerToken` annotation parameters
* **`@Timed` micrometer instrumentation** ‚Äî metrics recording overhead on every validation
* At 20¬µs, CDI overhead is negligible relative to the 1,060¬µs JWT validation cost
