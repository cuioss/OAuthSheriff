== Executive Summary

**Analysis Date**: March 2026 +
**Primary Finding**: RSA signature verification costs **1,060µs** under production-like concurrency (Mandrel native image, 50 connections). In JMH isolation the same operation takes 68µs — a ~14x concurrency multiplier whose individual causes are not decomposed.

== RSA Performance Under GraalVM Native

=== Integration Measurement (WRK, 50 connections)

The link:../../../benchmarking/doc/Analysis-03.2026-Latency-Decomposition.adoc[ablation study] isolates JWT validation cost at **1,060µs** (46.1% of the 2.30ms total request time). RSA signature verification (RS256) dominates this — it accounts for 88% of the library's processing pipeline.

=== JMH Isolation Comparison

In single-threaded JMH micro-benchmarks (no HTTP stack, no concurrency), the same RS256 operation takes **68µs**. The ~14x gap under concurrency is measured but **the exact causes are not individually decomposed**. See link:../../../benchmarking/doc/Analysis-03.2026-Integration.adoc#_root_cause_jwt_validation_under_concurrency[root cause analysis].

=== JFR CPU Profiling

From link:../../../benchmarking/doc/Analysis-03.2026-JFR-Profiling.adoc[JFR Profiling Analysis] (48,027 execution samples under load):

* **TLS Encryption**: 18.7% of CPU — shared infrastructure cost, not JWT-specific
* **CUI Sheriff OAuth Library**: 0.3% of CPU
* **BigInteger (RSA)**: 0.1% of CPU (46 samples)

NOTE: Low CPU% does not mean low wall-clock cost. The ablation study measures JWT validation at 46.1% of request time. The CPU/latency mismatch is documented but **the exact cause is unknown**. See link:../../../benchmarking/doc/Analysis-03.2026-JFR-Profiling.adoc#_why_cpu_doesnt_match_latency[JFR analysis].

== GraalVM Native Image Characteristics

* **No JIT Compilation**: Static AOT compilation
* **Predictable Performance**: No warmup period, consistent latency
* RSA verification: 1,060µs under concurrency (68µs in JMH isolation)

=== GraalVM 25 / Mandrel jdk-25 Improvements

* **WP-SCCP**: Whole-Program Sparse Conditional Constant Propagation (default)
* **ML Profile Inference**: 1-3% performance boost
* **JCMD Support**: Runtime diagnostics via jcmd
* **Vector API**: Initial optimization support

== Algorithm Comparison

[cols="2,2,2,3", options="header"]
|===
| Algorithm | Type | Measured Time (native, 50 conns) | Assessment

| RS256 | RSA 2048-bit | 1,060µs (68µs in JMH isolation) | Current default
| ES256 | ECDSA P-256 | — | Not benchmarked
| Ed25519 | EdDSA Curve25519 | — | Not benchmarked
|===

No alternative algorithm has been benchmarked. Algorithm choice should be driven by standards compliance, key management, and token size rather than performance.

== JCA Provider Landscape

* **Default JDK** (Recommended): 1,060µs RSA under concurrency, full native image support, automatic updates
* **BouncyCastle**: Only if specific algorithms needed; requires reflection configuration
* **Conscrypt**: Not suitable for GraalVM native image (BoringSSL dependencies)
* **Jipher**: No longer bundled with GraalVM 25; consider only for FIPS compliance

== Performance Monitoring

[source,bash]
----
# Build with monitoring support
native-image --enable-monitoring=jcmd,jfr,heapdump YourApp

# Runtime JFR recording
jcmd <pid> JFR.start duration=60s filename=profile.jfr
jcmd <pid> JFR.dump filename=profile.jfr
----
