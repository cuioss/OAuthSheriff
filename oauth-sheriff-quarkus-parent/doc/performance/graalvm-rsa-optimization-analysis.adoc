== Executive Summary

**Analysis Date**: March 2026 +
**Primary Finding**: RSA signature verification at 68µs (P50) accounts for 88% of the 77µs library validation time but only 0.1% of CPU samples under integration load.

== RSA Performance

=== JMH Benchmark Results

[cols="2,2,2", options="header"]
|===
| Operation | Time (P50) | Notes

| RS256 Signature Verification | 68µs | Default JDK provider
| Full JWT Parsing + Verification | 77µs | Includes JSON parsing
| Token Cache Lookup | 0.1µs | Lock-free concurrent cache
|===

See link:../../../benchmarking/doc/Analysis-03.2026-Micro.adoc[Microbenchmark Analysis] for full pipeline breakdown.

=== JFR CPU Profiling

From link:../../../benchmarking/doc/Analysis-03.2026-JFR-Profiling.adoc[JFR Profiling Analysis] (48,027 execution samples under load):

* **TLS Encryption**: 18.7% of CPU — shared infrastructure cost, not JWT-specific
* **CUI Sheriff OAuth Library**: 0.3% of CPU
* **BigInteger (RSA)**: 0.1% of CPU (46 samples)

NOTE: Low CPU% does not mean low wall-clock cost. The ablation study measures JWT validation at 46.1% of request time (1,060µs). The CPU/latency mismatch is documented but **the exact cause is unknown**. See link:../../../benchmarking/doc/Analysis-03.2026-JFR-Profiling.adoc#_why_cpu_doesnt_match_latency[JFR analysis].

== GraalVM Native Image Characteristics

* **No JIT Compilation**: Static AOT compilation
* **Predictable Performance**: No warmup period, consistent latency
* Default JDK RSA provider achieves 68µs in JMH isolation; under concurrency (50 connections) the ablation study measures 1,060µs for JWT validation — see link:../../../benchmarking/doc/Analysis-03.2026-Integration.adoc#_root_cause_jwt_validation_under_concurrency[root cause analysis]

=== GraalVM 25 / Mandrel jdk-25 Improvements

* **WP-SCCP**: Whole-Program Sparse Conditional Constant Propagation (default)
* **ML Profile Inference**: 1-3% performance boost
* **JCMD Support**: Runtime diagnostics via jcmd
* **Vector API**: Initial optimization support

== Algorithm Comparison

[cols="2,2,2,3", options="header"]
|===
| Algorithm | Type | Measured Time | Assessment

| RS256 | RSA 2048-bit | 68µs (JMH) / 1,060µs (integration) | Current default
| ES256 | ECDSA P-256 | — | Not benchmarked
| Ed25519 | EdDSA Curve25519 | — | Not benchmarked

No alternative algorithm has been benchmarked. Algorithm choice should be driven by standards compliance, key management, and token size rather than performance.

== JCA Provider Landscape

* **Default JDK** (Recommended): 68µs RSA in isolation (1,060µs under concurrency), full native image support, automatic updates
* **BouncyCastle**: Only if specific algorithms needed; requires reflection configuration
* **Conscrypt**: Not suitable for GraalVM native image (BoringSSL dependencies)
* **Jipher**: No longer bundled with GraalVM 25; consider only for FIPS compliance

== Performance Monitoring

[source,bash]
----
# Build with monitoring support
native-image --enable-monitoring=jcmd,jfr,heapdump YourApp

# Runtime JFR recording
jcmd <pid> JFR.start duration=60s filename=profile.jfr
jcmd <pid> JFR.dump filename=profile.jfr
----
