== Purpose

JWT validation performance characteristics, baselines, and analysis for the CUI OAuth Sheriff Quarkus integration.

== Performance Baselines (January 2026)

=== System Configuration

* **Platform**: Apple M4, 10 CPU cores, 32 GB RAM
* **Runtime**: Mandrel jdk-25 native image
* **Framework**: Quarkus 3.32.1
* **Container**: Docker with Keycloak for token issuance

=== Current Performance Summary

[cols="3,2,2,2", options="header"]
|===
| Endpoint | Throughput | P50 Latency | P99 Latency

| **JWT Validation** | 22,700-24,300 ops/s | 1.91ms | 4.78ms
| **Health Check** | 64,900-89,900 ops/s | 0.426-0.510ms | 1.37ms
| **JWT Library (JMH)** | 159,100 ops/s | 77us | N/A
|===

JWT validation throughput remains stable at 22.7-24.3K ops/s across 50-300 concurrent connections with a constant JWT/Health P50 ratio of 4.55x (SD 0.09).

== Latency Decomposition

[cols="2,2,2", options="header"]
|===
| Component | Latency (P50) | Percentage

| Shared Infrastructure (Docker, TLS, HTTP/2, Quarkus routing) | 0.510ms | 27%
| Framework Overhead (POST, JAX-RS, Jackson) | 0.313ms | 16%
| **Token Validation** (CDI producer + JWT validation) | **1.087ms** | **57%**
|===

**Total**: 0.510 + 0.313 + 1.087 = **1.91ms** (JWT P50 at 50 connections)

A mock JWT endpoint isolates the token validation cost at 1.087ms. The 14x gap between library (77us JMH) and integration (1.087ms) is caused by CDI `Instance.get()` resolution, InjectionPoint introspection, `@Timed` instrumentation, and object allocation. This overhead is **inherent to the security design** -- per-request authorization with different requirements per injection point. `@WithCaching` would cause a security bypass.

== JWT Library Performance (JMH Analysis)

[cols="2,2,2", options="header"]
|===
| Component | Time | Share of JWT Processing

| RSA Signature Verification | 68us | **88%**
| Token Parsing | 4.3us | 6%
| Cache Operations | 0.1us | <1%
| Other | ~4.6us | 6%
|===

RSA at 68us is NOT a bottleneck. JFR confirms BigInteger.modPow() accounts for only 0.1% of CPU samples.

== Algorithm Performance Characteristics

[cols="2,2,2,3", options="header"]
|===
| Algorithm | Type | Security Level | Performance Notes

| RS256/384/512 | RSA | 2048-bit | 68us - Current default, excellent
| ES256/384/512 | ECDSA | 256-384-bit curves | Expected faster, marginal benefit
| Ed25519/Ed448 | EdDSA | Curve25519/448 | Fastest, smaller signatures
|===

No urgent algorithm migration required. Algorithm choice should be driven by standards compliance, token size, and key management.
